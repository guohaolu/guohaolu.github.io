Java并发集合-并发哈希结构设计

并发哈希结构

设计一个支持并发访问的哈希结构（Concurrent Hash Structure）是一项挑战性任务，因为它需要在保证高性能的同时，还要确保数据的一致性和安全性。下面是一些设计并发哈希结构的关键考虑因素和实现策略。

设计目标

  1. 高性能：并发哈希结构应当支持高并发的读写操作，减少锁的竞争，提高整体性能。
  2. 线程安全：在多线程环境下，需要确保数据的一致性和安全性，避免数据竞争和死锁。
  3. 可扩展性：随着数据量的增长，哈希结构应当能够平滑地扩展，支持更多的数据和更高的并发度。
  4. 易用性：设计应当简单易用，易于理解和实现。

设计策略

1. 分段锁（Segment Locking）

  • 原理：将哈希表分成多个段（Segment），每个段有自己的锁。这样，在同一时刻，多个线程可以并发访问不同的段，减少了锁的竞争。
  • 实现：可以使用一个数组来存储多个哈希表，每个哈希表对应一个段，并且每个段有自己的锁。

2. 细粒度锁（Fine-Grained Locking）

  • 原理：在哈希表中，对单个桶（Bucket）进行加锁，而不是整个表。这样可以进一步减少锁的竞争。
  • 实现：每个桶都有自己的锁，当访问或修改该桶时，才对其加锁。

实现细节

  • 锁必须为可重入锁，当线程进行复杂操作的时候，不会死锁。锁同时也需要为读写锁，不会阻塞只读操作。
  • 集合大小在新增元素的时候需要自增，选用原子类型来支撑。
  • 扩容策略，设置一个固定大小的负载因子（平均每个桶存储的元素个数）。
  • add 方法新增元素后会检测集合是否满足扩容条件，若满足则进行扩容操作。扩容操作时不允许集合中存在写操作。

boolean add(T entry) 新增元素
void acquire(T entry) 获得锁
void release(T entry) 释放锁
void resize() 扩容
boolean policy() 检测是否达到扩容条件

随机过程

随机过程：随机变量的序列，如果我们固定某一个观察时刻t，则事物在时刻t出现的状态是随机的。

伯努利过程：
$$
X = { x_1, x_2, \ldots } = { x_n : n = 1, 2, \ldots ; x_n \in {1, 0} }
$$
从时间和状态空间的维度上，随机过程可以继续被划分：

  1. 离散时间：如果参数 n 取值于离散集，比如整数集Z 或非负整数集 N，则该过程是一个离散时间随机过程。
  2. 连续时间：如果参数n可以取任意实数值，则称其为连续时间随机过程。

对于每个随机变量 **xn**：

  1. 离散状态空间：如果每个随机变量 **xn**只能取有限个或可数无限个值，那么这个随机过程就称为离散状态空间随机过程。例如，伯努利过程中的随机变量只能取0或1两个值。
  2. 连续状态空间：如果每个随机变量 xn 可以取任意实数值或更高维度的值，那么就是连续状态空间随机过程。

马尔可夫链

马尔可夫链（Markov Chain）是一种特殊的随机过程，它在离散时间上进行，并且具有一个重要的特性——马尔可夫性质。

马尔可夫性质指的是，在给定当前状态的情况下，未来的状态仅依赖于当前状态，而不依赖于过去的历史状态。换句话说，一旦当前状态确定了，过去的状态就不再提供额外的信息来预测未来。

  • 状态空间S：可以是有限或无限。

  • 初始状态分布：马尔可夫链开始于某个初始状态的概率分布 π0。

  • 转移概率分布

时间齐次的马尔可夫链：转移概率分布与 t 无关。

转移概率矩阵

如状态空间 S 中仅有三种状态，则转移概率矩阵为 3*3 矩阵，元素定义为由一种状态转为另一种状态的概率。这样只要有了转移概率矩阵，我们就可以从一个过往的概率分布链式拓展下去。

Netty-核心API

ChannelHandler

处理 I/O 事件或拦截 I/O 操作，并将其转发到其 ChannelPipeline 中的下一个处理程序。

功能

  1. 处理I/O事件：
    • ChannelHandler可以处理各种I/O事件，如连接建立、断开、读取、写入等。
    • 通过实现ChannelInboundHandler或ChannelOutboundHandler接口，你可以编写自定义的ChannelHandler来处理这些事件。
  2. 读取和写入数据：
    • ChannelHandler可以读取从网络上接收的数据，并将其处理后写回网络。
    • 你可以实现channelRead()方法来处理读取的数据，并实现write()方法来发送数据。
  3. 异常处理：
    • ChannelHandler可以处理在通信过程中发生的异常情况。
    • 你可以实现exceptionCaught()方法来捕获和处理异常。
  4. 生命周期管理：
    • ChannelHandler可以参与到Channel的生命周期管理中。
    • 你可以实现channelRegistered()、channelActive()、channelInactive()等方法来处理Channel的生命周期事件。
  5. 自定义处理逻辑：
    • ChannelHandler可以实现自定义的业务逻辑，如协议解析、编码解码、安全认证等。
    • 你可以实现channelRead0()方法来处理具体的业务逻辑。

常用方法

以下是一些常用的ChannelHandler方法：

  1. 入站方法：
    • channelRegistered(ChannelHandlerContext ctx)：当Channel注册到EventLoop上时调用。
    • channelActive(ChannelHandlerContext ctx)：当Channel准备就绪，可以进行读写操作时调用。
    • channelInactive(ChannelHandlerContext ctx)：当Channel变得不可用时调用。
    • channelRead(ChannelHandlerContext ctx, Object msg)：当从网络上读取到消息时调用。
    • channelReadComplete(ChannelHandlerContext ctx)：当读取操作完成时调用。
    • userEventTriggered(ChannelHandlerContext ctx, Object evt)：当用户触发的事件发生时调用。
    • channelWritabilityChanged(ChannelHandlerContext ctx)：当Channel的可写状态发生变化时调用。
    • exceptionCaught(ChannelHandlerContext ctx, Throwable cause)：当发生异常时调用。
  2. 出站方法：
    • write(ChannelHandlerContext ctx, Object msg, Promise promise)：当需要写入数据到网络时调用。
    • flush(ChannelHandlerContext ctx)：当需要刷新输出缓冲区时调用。

ChannelPipeline

由多个 ChannelHandler 组成的链为ChannelPipeline。

  • InboundHandler 按照 Pipeline 正序处理入站，OutboundHandler 按照 Pipeline 逆序处理出站。

Java-网络编程模式

数据传输过程

  1. 数据到达网络设备：
    • 当数据包从网络到达时，网卡（NIC，Network Interface Card）接收到这些数据包。
  2. DMA操作：
    • 网卡通过 DMA 操作将接收到的数据包直接写入预先分配好的内核空间缓冲区。DMA 操作允许数据直接从网卡传输到内存，而不需要 CPU 的介入，从而减轻了 CPU 的负担。
  3. 网卡驱动程序的作用：
    • 网卡驱动程序负责配置 DMA 操作，并管理内核空间缓冲区。
    • 驱动程序还负责检测 DMA 操作的完成情况，并通知内核数据已经就绪。
    • 一旦 DMA 操作完成，驱动程序会通知操作系统内核，告诉内核哪些缓冲区中有新的数据。
  4. 内核处理数据：
    • 内核收到通知后，会处理这些数据，例如将数据从 DMA 缓冲区复制到应用程序可以访问的内核缓冲区。
    • 内核还会设置相应的文件描述符（在 Java NIO 中对应Channel）的状态，表示数据已经就绪，可以被应用程序读取。
  • DMA：DMA 允许网卡直接将数据写入内核空间缓冲区，从而提高数据传输效率。
  • 网卡驱动程序：驱动程序负责配置 DMA 操作，并管理内核空间缓冲区。它还负责通知内核数据已经就绪。

高级IO功能

  1. 文件锁定：
    • 文件锁定可以防止多个进程或线程同时修改同一个文件，从而保证数据的一致性和完整性。
    • 传统的 Java I/O 机制缺乏对文件锁定的支持，容易导致并发访问时的数据冲突。
  2. 非阻塞I/O：
    • 非阻塞 I/O 允许应用程序在没有数据可读或可写时立即返回，而不是阻塞等待。这样可以提高系统的并发处理能力。
    • 传统的 Java I/O 机制通常采用阻塞式 I/O，无法充分利用现代操作系统的非阻塞 I/O 功能。
  3. 就绪性选择：
    • 就绪性选择（如select、poll、epoll等）允许应用程序监控多个文件描述符的状态，当某个文件描述符上的数据可读或可写时，可以立即进行处理。
    • 传统的 Java I/O 机制缺乏对就绪性选择的支持，无法高效地处理多个并发连接。
  4. 内存映射：
    • 内存映射允许将文件或设备直接映射到内存中，从而提高数据访问速度。
    • 传统的 Java I/O 机制通常采用逐字节读写的方式，无法充分利用内存映射的优势。

BIO

在 Java 的 BIO（Blocking I/O）模型中，很多操作都是同步阻塞的。常见的阻塞操作：

  1. **接受连接 (accept)**：
    • ServerSocket.accept() 用来等待客户端的连接请求。如果没有客户端连接，这个方法会一直阻塞，直到有一个新的连接到达。
  2. **读取数据 (read)**：
    • InputStream.read() 用来从客户端读取数据。如果没有数据可读，这个方法会阻塞，直到有数据可读。
    • BufferedReader.readLine() 同样会阻塞，直到一行数据读取完毕或到达流的末尾。
  3. **写入数据 (write)**：
    • OutputStream.write() 用来向客户端发送数据。在某些情况下，如果网络拥塞或者客户端缓冲区满，这个方法也可能暂时阻塞。

多线程BIO的缺点：

  • 资源开销大，降低了服务端的吞吐量。

NIO

NIO：基于 IO 多路复用的同步非阻塞 IO。

在 Java NIO（Non-blocking I/O）中，使用 IO 多路复用技术可以实现同步非阻塞的 IO 操作。具体来说：

  • IO多路复用：一个线程可以管理（轮询：select，poll）多个 Socket 连接。当某个 Socket 的缓冲区（内核空间）中有数据可读或可写时，Selector 会通知（epoll）对应的 IO 线程进行读写操作。这样，一个线程可以同时监控多个 Socket 连接，极大地提高了系统的并发处理能力。
  • IO操作时会阻塞线程，但此时数据已经就绪：当 Selector 通知某个 Socket 连接的数据已经就绪时，线程会执行读写操作。虽然这些读写操作本身是同步的，但由于数据已经就绪（即数据已经存在于内核空间的缓冲区中），因此读写操作不会长时间阻塞线程。这种机制使得每次读写操作都非常高效，大大提升了整体的系统性能。

tips：read 操作本身仍然是阻塞的，因为它需要等待数据从内核缓冲区复制到用户缓冲区。

AIO

异步非阻塞 IO，应用较少，主要是 Linux 上的异步 IO 实现比较特别。

主从Reactor模式

1. 主Reactor

  • 职责：
    • 监听新连接的到来。
    • 接受新连接，并将其分发给从 Reactor 进行处理。
  • 具体操作：
    • 创建一个Selector实例，并将ServerSocketChannel注册到该Selector上，监听SelectionKey.OP_ACCEPT事件。
    • 在Selector中循环等待新连接的到来。
    • 当ServerSocketChannel上的SelectionKey.OP_ACCEPT事件就绪时，接受新连接，并将新连接的SocketChannel注册到一个从Reactor的Selector上，监听SelectionKey.OP_READ事件。

2. 从Reactor

  • 职责：
    • 监控已建立的连接上的读写事件。
    • 处理具体的读写操作。
    • 将读写操作分发给 Worker 线程进行处理（如果有 Worker 线程）。
  • 具体操作：
    • 创建一个Selector实例，并将已接受的新连接的SocketChannel注册到该Selector上，监听SelectionKey.OP_READ事件。
    • 在Selector中循环等待读写事件的到来。
    • 当SocketChannel上的SelectionKey.OP_READ事件就绪时，读取数据，并将数据交给 Worker 线程进行处理（如果有 Worker 线程）。
    • 当SocketChannel上的SelectionKey.OP_WRITE事件就绪时，写入数据。

3. Worker线程

  • 职责：
    • 执行具体的业务逻辑。
    • 处理来自客户端的数据。
    • 准备响应数据，并将响应数据交给从 Reactor 进行写操作。
  • 具体操作：
    • 接收从 Reactor 传递过来的数据。
    • 处理数据，生成响应数据。
    • 将响应数据交给从 Reactor，由从 Reactor 写回到客户端。

Netty线程模型

BossGroup

职责：

  • 负责接收新进来的连接请求。
  • 一旦接收到新的连接请求，BossGroup会创建一个新的Channel并将其注册到WorkerGroup中的某个EventLoop上。

具体操作：

  1. 监听新连接
    • BossGroup中的每个EventLoop都会监听新连接的到来。通常，BossGroup只包含一个EventLoop，因为只需要一个线程来监听新连接。
  2. 处理新连接
    • 当新的连接请求到达时，BossGroup会创建一个新的Channel（例如SocketChannel）并将其注册到WorkerGroup中的某个EventLoop上。
    • 这个新的Channel将由WorkerGroup中的EventLoop来处理后续的I/O操作，包括读取数据、写入数据等。

WorkerGroup

职责：

  • 负责处理已经被接收的连接的所有 I/O 操作，包括读取数据、写入数据等。
  • 可以包含多个EventLoop，每个EventLoop都可以独立地处理多个Channel上的I/O事件。

具体操作：

  1. 处理 I/O 事件
    • WorkerGroup中的每个EventLoop都会处理分配给它的Channel上的 I/O 事件。
    • 每个EventLoop都有一个自己的事件循环，可以独立地处理多个Channel上的事件。
  2. 读取和写入数据
    • 当Channel上的数据可读或可写时，对应的EventLoop会处理这些事件。
    • EventLoop会调用相应的ChannelHandler来读取或写入数据。
