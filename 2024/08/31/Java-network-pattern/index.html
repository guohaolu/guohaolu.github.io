Java-网络编程模式

发表于 2024-08-31

数据传输过程

  1. 数据到达网络设备：
    • 当数据包从网络到达时，网卡（NIC，Network Interface Card）接收到这些数据包。
  2. DMA操作：
    • 网卡通过DMA操作将接收到的数据包直接写入预先分配好的内核空间缓冲区。DMA操作允许数据直接从网卡传输到内存，而不需要CPU的介入，从而减轻了CPU的负担。
  3. 网卡驱动程序的作用：
    • 网卡驱动程序负责配置DMA操作，并管理内核空间缓冲区。
    • 驱动程序还负责检测DMA操作的完成情况，并通知内核数据已经就绪。
    • 一旦DMA操作完成，驱动程序会通知操作系统内核，告诉内核哪些缓冲区中有新的数据。
  4. 内核处理数据：
    • 内核收到通知后，会处理这些数据，例如将数据从DMA缓冲区复制到应用程序可以访问的内核缓冲区。
    • 内核还会设置相应的文件描述符（在Java NIO中对应Channel）的状态，表示数据已经就绪，可以被应用程序读取。
  • DMA：DMA允许网卡直接将数据写入内核空间缓冲区，从而提高数据传输效率。
  • 网卡驱动程序：驱动程序负责配置DMA操作，并管理内核空间缓冲区。它还负责通知内核数据已经就绪。

高级IO功能

  1. 文件锁定：
    • 文件锁定可以防止多个进程或线程同时修改同一个文件，从而保证数据的一致性和完整性。
    • 传统的Java I/O机制缺乏对文件锁定的支持，容易导致并发访问时的数据冲突。
  2. 非阻塞I/O：
    • 非阻塞I/O允许应用程序在没有数据可读或可写时立即返回，而不是阻塞等待。这样可以提高系统的并发处理能力。
    • 传统的Java I/O机制通常采用阻塞式I/O，无法充分利用现代操作系统的非阻塞I/O功能。
  3. 就绪性选择：
    • 就绪性选择（如select、poll、epoll等）允许应用程序监控多个文件描述符的状态，当某个文件描述符上的数据可读或可写时，可以立即进行处理。
    • 传统的Java I/O机制缺乏对就绪性选择的支持，无法高效地处理多个并发连接。
  4. 内存映射：
    • 内存映射允许将文件或设备直接映射到内存中，从而提高数据访问速度。
    • 传统的Java I/O机制通常采用逐字节读写的方式，无法充分利用内存映射的优势。

BIO

在Java的BIO（Blocking I/O）模型中，很多操作都是同步阻塞的。常见的阻塞操作：

  1. **接受连接 (accept)**：
    • ServerSocket.accept() 用来等待客户端的连接请求。如果没有客户端连接，这个方法会一直阻塞，直到有一个新的连接到达。
  2. **读取数据 (read)**：
    • InputStream.read() 用来从客户端读取数据。如果没有数据可读，这个方法会阻塞，直到有数据可读。
    • BufferedReader.readLine() 同样会阻塞，直到一行数据读取完毕或到达流的末尾。
  3. **写入数据 (write)**：
    • OutputStream.write() 用来向客户端发送数据。在某些情况下，如果网络拥塞或者客户端缓冲区满，这个方法也可能暂时阻塞。

多线程BIO的缺点：

  • 资源开销大，降低了服务端的吞吐量。

NIO

NIO：基于IO多路复用的同步非阻塞IO。

在Java NIO（Non-blocking I/O）中，使用IO多路复用技术可以实现同步非阻塞的IO操作。具体来说：

  • IO多路复用：一个线程可以管理（轮询：select，poll）多个Socket连接。当某个Socket的缓冲区（内核空间）中有数据可读或可写时，Selector会通知（epoll）对应的IO线程进行读写操作。这样，一个线程可以同时监控多个Socket连接，极大地提高了系统的并发处理能力。
  • IO操作时会阻塞线程，但此时数据已经就绪：当Selector通知某个Socket连接的数据已经就绪时，线程会执行读写操作。虽然这些读写操作本身是同步的，但由于数据已经就绪（即数据已经存在于内核空间的缓冲区中），因此读写操作不会长时间阻塞线程。这种机制使得每次读写操作都非常高效，大大提升了整体的系统性能。

tips：read操作本身仍然是阻塞的，因为它需要等待数据从内核缓冲区复制到用户缓冲区。

主从Reactor模式

1. 主Reactor

  • 职责：
    • 监听新连接的到来。
    • 接受新连接，并将其分发给从Reactor进行处理。
  • 具体操作：
    • 创建一个Selector实例，并将ServerSocketChannel注册到该Selector上，监听SelectionKey.OP_ACCEPT事件。
    • 在Selector中循环等待新连接的到来。
    • 当ServerSocketChannel上的SelectionKey.OP_ACCEPT事件就绪时，接受新连接，并将新连接的SocketChannel注册到一个从Reactor的Selector上，监听SelectionKey.OP_READ事件。

2. 从Reactor

  • 职责：
    • 监控已建立的连接上的读写事件。
    • 处理具体的读写操作。
    • 将读写操作分发给Worker线程进行处理（如果有Worker线程）。
  • 具体操作：
    • 创建一个Selector实例，并将已接受的新连接的SocketChannel注册到该Selector上，监听SelectionKey.OP_READ事件。
    • 在Selector中循环等待读写事件的到来。
    • 当SocketChannel上的SelectionKey.OP_READ事件就绪时，读取数据，并将数据交给Worker线程进行处理（如果有Worker线程）。
    • 当SocketChannel上的SelectionKey.OP_WRITE事件就绪时，写入数据。

3. Worker线程

  • 职责：
    • 执行具体的业务逻辑。
    • 处理来自客户端的数据。
    • 准备响应数据，并将响应数据交给从Reactor进行写操作。
  • 具体操作：
    • 接收从Reactor传递过来的数据。
    • 处理数据，生成响应数据。
    • 将响应数据交给从Reactor，由从Reactor写回到客户端。

Netty线程模型

BossGroup

职责：

  • 负责接收新进来的连接请求。
  • 一旦接收到新的连接请求，BossGroup会创建一个新的Channel并将其注册到WorkerGroup中的某个EventLoop上。

具体操作：

  1. 监听新连接
    • BossGroup中的每个EventLoop都会监听新连接的到来。通常，BossGroup只包含一个EventLoop，因为只需要一个线程来监听新连接。
  2. 处理新连接
    • 当新的连接请求到达时，BossGroup会创建一个新的Channel（例如SocketChannel）并将其注册到WorkerGroup中的某个EventLoop上。
    • 这个新的Channel将由WorkerGroup中的EventLoop来处理后续的I/O操作，包括读取数据、写入数据等。

WorkerGroup

职责：

  • 负责处理已经被接收的连接的所有I/O操作，包括读取数据、写入数据等。
  • 可以包含多个EventLoop，每个EventLoop都可以独立地处理多个Channel上的I/O事件。

具体操作：

  1. 处理I/O事件
    • WorkerGroup中的每个EventLoop都会处理分配给它的Channel上的I/O事件。
    • 每个EventLoop都有一个自己的事件循环，可以独立地处理多个Channel上的事件。
  2. 读取和写入数据
    • 当Channel上的数据可读或可写时，对应的EventLoop会处理这些事件。
    • EventLoop会调用相应的ChannelHandler来读取或写入数据。
